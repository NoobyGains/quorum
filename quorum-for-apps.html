Quorum for Apps

Cross-vendor adversarial review packaged for non-developers shipping real apps. Two different-vendor AIs don't just help build — they argue before anything lands. Every disagreement is a typed artifact you can scroll through, replay, and cite.

4-week MVP Next.js only at launch governance as moat measured against single-agent control

What changed in v2 — Codex blocked the first draft. Five substantive cuts.
  • Novelty: "cross-vendor review + audit trail" → narrower: "cross-vendor adversarial review for non-developers building real apps." Factory, OpenHands, Codebuff, Windsurf already claim vendor-agnostic or review-heavy workflows.
  • 3 templates (Next.js + WordPress + iOS) → Next.js only. iOS is a week-one trap (Xcode / signing overhead); WordPress is a later vertical.
  • Audience: "vibe coders" → founders & operators already burned by silent AI failure, wanting receipts + second opinions. Pure non-devs are downstream.
  • Success metric: "<2h to ship" → rework / defect-escape rate vs. a single-agent control. Shipping fast proves scaffolding; beating the single-agent baseline proves the product.
  • Five MVP components → three. Standalone web UI and decision-log viewer deferred to v1.1; artifact logging stays but runs behind the scenes.

MVP components — three kept, two deferred

01 · kept

One-command install

npm create quorum-app my-thing. Scaffolds git, installs MCP, wires both agents. Zero CLI thereafter.

02 · kept

Next.js template with review loop

One opinionated web template. Conventions + test stubs so both agents agree what "done" means. Artifacts log behind the scenes.

03 · kept

Hard spend cap + kill-switch

Soft-warn at target, hard-stop at 2× with checkpoint-and-resume. Hygiene, not differentiation.

04 · deferred

Standalone web UI

Live Plan→Review loop visualization. Moves to v1.1. MVP uses existing CLI + VS Code.

05 · deferred

Decision-log viewer

Timeline of Plans/Reviews/Decisions. Moves to v1.1. Artifacts still captured — we just don't ship a browser for them at launch.

WEEK 4

Success metric — measured against single-agent control

Hypothesis. Feature requests routed through two-vendor adversarial review produce materially fewer rework events and defect escapes than the same requests routed through a single agent — at acceptable latency and cost overhead.
Primary outcome

48-hour post-ship rework rate. Cross-vendor flow must be ≤ 50% of the single-agent control's rate on matched tasks to justify the cost.

Secondary outcome

User willingness to approve a Plan without opening the code. Directly measures whether the review signal is trusted.

Guardrail

Latency and $ per shipped feature stay within 2× of the single-agent baseline. Past that, cross-vendor review is a research toy, not a product.

What would kill the thesis

If rework rate matches the single-agent control on tasks of meaningful complexity, the differentiator is ornamental and we pivot before week 4 ends.

Moat — governance, not generation

L0-L2 · OSS
Open-source the artifact protocol, git-ref scheme, and MCP server. Adoption + trust compound when anyone can read the format and run their own store.
L3-L4 · OSS
Open-source the watchdog daemon, disagreement protocol, and CLI. Lock-in here would kill community adoption and it isn't where the money is anyway.
L5-L7 · commercial
Paid managed control plane: approval policy rules, reviewer-pool routing, org-scoped cost ceilings, exportable compliance evidence, hosted dashboards, SSO + audit log retention. Sold to the burned founder and the enterprise buyer that comes after.

The copyable part (ask a second model) is the L2 primitive. The defensible part is the policy layer that turns it into something a compliance officer can sign off on.

Risks we're tracking — updated severity

high · market

Replit Agent and GitHub Spark already sell non-coders a build-test-refine loop. Factory, OpenHands, Codebuff, Windsurf lean into review-heavy or vendor-agnostic workflows. We must prove cross-vendor review changes outcomes, not just process.

high · thesis

The load-bearing claim is that two vendors disagreeing catches things one vendor doesn't. If the outcome benchmark shows parity on real tasks, the product is ornamental. Kill the program fast in that case.

medium · user

Burned-founder hypothesis is plausible but unvalidated. Pre-ship signal: do prospects spontaneously ask for a second model after a failure? Tolerate slower runs for higher confidence? Pick Quorum over a single-agent control in repeated tests?

medium · cost

Two-vendor review is intrinsically slower and pricier than single-agent. Users must perceive the rework-avoidance savings as worth more than the latency + $ cost. If perception lags reality, we lose.

low · scope

One template is credible in 4 weeks. Biggest residual execution risk is the measurement harness in week 3 — falling back to anecdotal evidence is acceptable if the harness slips but weakens the narrative.
